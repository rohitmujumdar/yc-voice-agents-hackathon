The bug we just fixed

When Cekura makes a WebRTC call, Pipecat Cloud hands the bot a DailySessionArguments object. The bot's match runner_args block only handled SmallWebRTCRunnerArguments (local browser) and WebSocketRunnerArguments (Twilio). Daily args hit the catch-all and the bot exited silently. So the container was alive but had no audio transport. The test agent said "Hello", got nothing back, and timed out after 10 seconds. Fix: added daily to Pipecat extras and a DailyTransport case to the bot.

The mental model

  • Pipecat is the framework that wires audio in → STT → LLM → TTS → audio out. It does NOT decide what Cora says, that's the LLM.
  • Pipecat Cloud is the hosting service that runs your bot containers. It uses Daily.co under the hood for WebRTC.
  • Twilio is only used when a real phone calls your number. Cekura uses the WebRTC path, not Twilio.
  • NVIDIA Nemotron is the open-weight LLM and STT, hosted on AWS for the hackathon.
  • Gradium is the TTS (turns Cora's words into audio).
  • Cekura simulates callers, scores transcripts, and tells you what's failing.
  • The "product" you built is the Python code in bot-cora.py + clinic_backend.py: the system prompt, the tools, the mock data. Everything else is infrastructure.
