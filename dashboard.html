The Story

Each Cekura iteration found a different class of failure that local development never surfaced. Iteration 1: no audio transport (DailySessionArguments unhandled). Iteration 2: LLM and pipeline disagreed on a turn-completion token — Pipecat was waiting for a checkmark Nemotron never sent. Iteration 3: concurrency cap blocked half the parallel calls. Iteration 4: cold-start race condition. Iteration 5: hardened the prompt against the actual mistake patterns Cekura had surfaced. Plus 4 adversarial scenarios — clinic-realistic, not generic jailbreaks. Cora refused all of them. The loop never stops finding bugs. That's the point.

How we avoid overfitting the curve

  • Multi-metric scoring. Outcome alignment, latency, interruption rate, transcription accuracy, tool-call success. A "fix" that improves outcome but tanks latency would be a regression — none of the v1.1→v1.3 changes did that.
  • Generated diversity. Red-team scenarios were auto-generated by Cekura with sophisticated multi-turn social engineering, not hand-crafted to be easy wins.
  • Realistic attack patterns. All 4 red-team scenarios use authority (fake auditor, fake QA, VIP) and urgency, not "DAN" or "grandma" jailbreaks. They're attacks a real clinic might face.
  • Safety-critical changes need a human. Emergency-detection and PHI-handling rules require approval; the loop never silently changes them.
  • Production transcripts in scope. Cekura observability mode evaluates real call recordings — when judges call the live number, that call also gets logged with eval scores.

The loop, end to end

  1. Eval set in Cekura. 15 patient workflow scenarios + 4 adversarial red-team scenarios. Tags scope coverage: routing, info collection, emergency, escalation, attacks.
  2. Run against deployed agent. Cekura simulates 19 calls in parallel via WebRTC to Pipecat Cloud. Each call gets transcribed, scored, and stored.
  3. Diagnose failures. Cekura clusters failures by root cause: "the agent kept routing refills wrong", "the agent kept going silent after greeting", etc. Each cluster is one bug, not 15.
  4. Apply fix. Edit system prompt, redeploy. Often a single change patches a cluster.
  5. Re-run. Same 19 scenarios. Compare scores. Surface any regressions.
  6. Repeat until convergence. 4 iterations got us from 0% → 47% workflow + 100% red-team.
